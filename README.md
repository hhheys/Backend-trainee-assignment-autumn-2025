# Backend Trainee Assignment Autumn 2025

Репозиторий содержит решение задания для стажёра по backend на осень 2025 года.
Проект написан на Go и использует Docker и Docker Compose для упрощённого развёртывания.
API специфицировано через OpenAPI, линтинг и форматирование кода обеспечивается `golangci-lint`.

## Структура проекта

```
├── cmd/server/            # Точка входа приложения
├── internal/              # Бизнес-логика и вспомогательные пакеты
├── .env.example           # Пример конфигурации окружения
├── Dockerfile             # Сборка Docker-образа приложения
├── docker-compose.yaml    # Сборка и запуск контейнеров
├── .golangci.yaml         # Конфигурация линтера golangci-lint
├── go.mod / go.sum        # Модули Go
├── openapi.yml            # Спецификация API
└── task.md                # Описание задания
```

## Технологии

* Язык: Go
* API: OpenAPI (`openapi.yml`)
* Контейнеризация: Docker, Docker Compose
* Линтер: GolangCI-Lint
* Миграции: Golang-Migrate
* Драйвер базы данных: database/sql

## Быстрый старт

1. Клонируйте репозиторий:

```bash
git clone https://github.com/hhheys/Backend-trainee-assignment-autumn-2025.git
cd Backend-trainee-assignment-autumn-2025
```

2. Создайте файл окружения:

```bash
cp .env.example .env
# при необходимости отредактируйте переменные среды
```

3. Сборка и запуск через Docker Compose:

```bash
docker-compose up --build
```

4. Сервер будет доступен по адресу, указанному в `.env` (по умолчанию `http://localhost:8080`).

5. Документация API доступна через файл `openapi.yml`.


## Реализация Основной Логики

### Создание сущности (Create)
При создании новой сущности сервис:
1. Выполняет валидацию входных данных.
2. Проверяет наличие связанных сущностей при необходимости.
3. Формирует доменную модель и передаёт её в репозиторий.
4. Возвращает созданный объект с системными полями (ID, timestamps).

### Обновление данных (Update)
1. Получение текущей версии объекта из базы.
2. Проверка прав пользователя при необходимости.
3. Обновление только разрешённых атрибутов.
4. Сохранение результата через репозиторий.

### Сложные бизнес-процессы
Если операция состоит из нескольких шагов:
- используется транзакция на уровне репозитория;
- выполняется последовательность бизнес-проверок;
- при любой ошибке транзакция откатывается.

## 3. Вопросы и Допущения

| Вопрос / Проблема                                                  | Допущение и Обоснование                                                                                                             |
|--------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------|
| **Механизм авторизации**                                           | Был использован примитивный способ авторизации через Access Token. Обоснование: минимальная сложность, не было уточнений в задании. |
| **Работа со временем**                                             | Используется UTC. Основание: отсутствие требований по таймзоне.                                                                     |
| **Уровень логирования**                                            | Включено базовое логирование ошибок и ключевых действий. Обоснование: минимально необходимый уровень наблюдаемости.                   |
| **Обработка ошибок**                                               | Введены внутренние ошибки с маппингом на HTTP-статусы (400, 404, 409, 500). Обоснование: предсказуемость API.                         |
| **Передача в роут создания команды некорректные id пользователей** | Создается команда с действительно существующими пользователями. Если все id были переданы некорректно, состав команды будет пустой. |
| **Выбор типа данных для id**                                       | Используется UUID. Обоснование: максимальная универсальность.                                                                         |
